| | |
|---|---|
| **Datei** | `app/api/setup/route.ts`, `middleware.ts` Zeile 36 |
| **Problem** | Route ist aus dem Auth-Matcher ausgenommen. Standardpasswort `admin123` ist hardcoded. | **Aufwand** | ~1 Stunde |

### Schwere 2 — Feature-Lücken (relevant für vollständigen Workflow)

#### FIX-04 · `retry.ts` verwendet `console.warn/error` statt Pino-Logger

| | |
|---|---|
| **Datei** | `lib/utils/retry.ts`, Zeilen 28, 35, 43 |
| **Problem** | Alle anderen Module nutzen `logger` (pino). Retry-Warnungen und -Fehler landen als unstrukturierter Console-Output — nicht in JSON-Logs. Inkonsistenz mit NFA-11. |
| **Aufwand** | ~15 Minuten |

#### FIX-05 · FA-F-05a: PDF/DOCX-Import für Positionierungsdokument fehlt

| | |
|---|---|

| **Aufwand** | 1–2 Tage (`pdfjs-dist` + `mammoth` bereits im Tech-Stack vorgesehen) |
| **Referenz** | FA-F-05a |

#### FIX-06 · AES-Verschlüsselung: Kein Versions-Präfix für Key-Rotation

| | |
|---|---|
| **Datei** | `lib/crypto/aes.ts` |
| **Problem** | ADR-003 dokumentiert dies als bekannte Lücke: Format ist `iv:authTag:ciphertext` ohne `v1:`-Präfix. Key-Rotation zu `ENCRYPTION_SECRET_V2` ist damit nicht möglich ohne alle verschlüsselten Werte gleichzeitig zu migrieren. |
| **Aufwand** | 1 Tag (Migration + Versionspräfix) |
| **Referenz** | ADR-003 |

### Schwere 3 — Qualitätsmängel

#### FIX-07 · Testabudeckung zu gering (NFA-16 partiell)

| | |
|---|---|
| **Befund** | 6 Unit-Test-Dateien, 1 Integration-Test, 1 E2E-Spec (nur Login). **Fehlt:** E2E URL→ZIP-Flow (roadmap Sprint 2 geplant aber nicht vorhanden), Unit-Tests für: Chat-Versioning, Cost-Aggregator, Dateinamen-Konvention, Template-Loader, WordPress-Formatter, KlickTipp-Formatter, Email-Templates. **0 API-Route-Tests** außer `/api/generate/start`. |
| **Aufwand** | 3–5 Tage |
| **Referenz** | NFA-16 |

#### FIX-08 · Audit-Log-Retention fehlt

| | |
|---|---|
| **Befund** | Plan/NFA spezifiziert "30 Tage lokal". Die `AuditLog`-Tabelle wächst unbegrenzt. Kein Cron-Job für Cleanup. Bei intensivem Betrieb (viele Generierungen, Exports) kann die Tabelle signifikant wachsen. |
| **Aufwand** | ~1 Stunde (Cron-Job in `scheduler.ts` ergänzen) |

#### FIX-09 · positioningDocument im Klartext in DB

| | |
|---|---|
| **Befund** | `Project.positioningDocument String?` — unverschlüsselt. Kann Praxis-sensible Informationen (Leistungsschwerpunkte, interne Positionierung) enthalten. NFA-03 (Envelope Encryption) ist erfüllt für API-Keys, aber nicht für dieses Feld. Als bekannte Abweichung in `forge-web-deviations.md` dokumentiert (Timeline Sprint 3). |
| **Empfehlung** | In Phase 5 migrieren. |

#### FIX-10 · In-Memory-Queue (queue.ts) nicht DB-backed

| | |
|---|---|
| **Befund** | `lib/generation/queue.ts` verwendet eine in-memory Set für laufende Jobs. Bei App-Neustart (Coolify auto-deploy) gehen Queue-Zustände verloren. Der einzelne `GenerationJob`-DB-Eintrag wird korrekt gespeichert, aber die Concurrent-Counter-Logik ist flüchtig. Kein Showstopper für Einzel-Instanz-Betrieb. |
| **Empfehlung** | Für Phase 5 / horizontales Scaling: Queue auf DB-Counter umstellen. |
